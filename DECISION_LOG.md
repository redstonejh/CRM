# Decision Log

| Component | Verdict | Reason |
|---|---|---|
| `ticketing` repository | Use as Phase 1 base | It already contains the canonical Electron shell, dashboard canvas, ticket backend seam, `ticket-stacks.js`, and `ticket-detail.js` called out in the plan. |
| `fractal-calendar.js` | Import unchanged for Phase 1 | The plan names the camera/calendar split for Phase 6; Phase 1 only needs the existing calendar experience present in the merged shell. |
| `ticket-detail.js` and `ticket-stacks.js` choreography | Keep active and behavior-compatible | The ticketing behavior is the regression surface for the later card-system factory refactor. |
| CRM identity strings | Rename now | The final repo should launch as CRM while retaining the ticketing seams for upcoming phases. |
| `name-and-info-cards` repository | Inspect, do not copy yet | Its People-specific changes belong to the Phase 5 module, after the card-system factory exists. |
| MQTT dependency and broker settings | Remove | The final CRM backend is Postgres/API; MQTT is not a retained architecture seam. |
| `electron/store.js` | Add API-backed seam | Existing `window.tickets` can run as an adapter while future `window.deals`, `window.contacts`, and other entity bridges share the same backend contract. |
| `server/` Postgres API | Add in Phase 2 | Shared team truth requires a central API, version checks, soft-delete fields, and a change stream instead of per-machine state. |
| `ticket-stacks.js` shared state | Move stage/rank/trash/meta writes toward record docs | Team CRM state cannot stay per-machine; local state now acts as optimistic fallback while the API-backed doc becomes shared truth. |
| `ticket-stacks.js` deck/bucket choreography | Extract to `card-system.js` factory | The drag, fan, stack, zone, trash, grid-drop, scroll, and focus mechanics are entity-blind; ticket assumptions are concentrated in source bridge, stages, field schema, card face, palette, storage keys, and verbs. |
| Ticket stage field schema | Keep as ticket config | Triage/investigation/resolution prompts are ticket workflow content, not card-system mechanics. They become the first factory config instead of remaining hard-coded choreography. |
| Ticket severity palette | Keep as ticket config | Deals and tasks can reuse the intensity idea, but contacts should not inherit incident severity styling. The palette belongs to each entity instance. |
| `ticket-detail.js` open/close motion | Extract to `card-detail.js` factory | Context-aware flyout, side selection, depth-of-field, clipping, and close choreography are reusable; the panel fields and save behavior are entity content. |
| Ticket detail panel fields | Keep as ticket config through stack API | The current panel shows only the active ticket stage fields and validates draft creation; future entities need different content without changing the motion engine. |
| Existing `window.ticketStacks` / `window.ticketDetail` globals | Preserve as ticket instance aliases | Ticketing is the regression baseline for Phase 3, so the renderer contract stays stable while the implementation moves behind factories. |
| Pipeline module | Instantiate through `card-system.js` / `card-detail.js` | Deals need the same deck, bucket, trash, grid-drop, and detail choreography, but with deal stages, temperature values, Won state, and deal-specific fields. |
| Tickets and Pipeline overlays | Add explicit module switch | The current card theater occupies one overlay plane; two active card-system instances would overlap. The switch keeps one active instance while preserving both configurations. |
| Pipeline right deck | Use as Won pile | The right deck's mechanics already represent a completed/reward pile; deals map that state to `state: "won"` instead of ticket `resolved`. |
| Drop-to-link | Add generic card-on-card drop hook, persist deal links as `relatedDealIds` | Relationship gestures belong in the card-system choreography; the deal module supplies the persistence rule so future Contacts/Companies can reuse the gesture without inheriting deal semantics. |
| People module | Instantiate Contacts through `card-system.js` / `card-detail.js` | The donor People repo proved the same card/deck choreography works for people, but its ticket fork is not copied; contacts become another factory instance on the shared Postgres/API entity bridge. |
| Contact glow/severity | Strip incident heat from contacts | Human relationship records should not inherit outage severity colours. People cards use a neutral card surface and small attention status instead of red/yellow/green heat. |
| People corner decks | Keep one attention/unbucketed deck plus recycle bin | The resolved/right pile is wrong for contacts. The left pile remains for contacts needing attention or assignment, while the right-side mechanics are retained only as the universal trash control. |
| People buckets | Use free company-relationship buckets | Company grouping is not a gated pipeline. Contacts can move freely among customer/prospect/partner/vendor buckets, with shared company/role/last-touch fields de-duplicated on the card face. |
| `fractal-calendar.js` camera mechanics | Extract to `fractal-camera.js` | The uniform zoom, warm expander, layer stack, resize layout, and back-navigation are not calendar-specific; Calendar and Home now consume the same camera factory. |
| Calendar year | Parameterize with year paging | The hard-coded `2026` limit was a content constraint, not camera behavior. Calendar keeps the same month/day math but stores the active year and exposes previous/next controls. |
| Calendar card drops | Persist `scheduledDate` on dropped records | Scheduling is a CRM relationship between a record and a day. Dropping grid cards onto day buckets updates the entity through the existing Postgres/API bridge without adding a broker or new renderer store. |
| Home menu | Build as a camera instance | Module navigation uses the same nested-bucket language as Calendar instead of a new UI species. Home buckets activate existing workspace modules through the shared switch. |
| Monitor `status-feed.js` | Leave stubbed for monitoring, add CRM report feed beside it | The old feed is circuit-specific and would reintroduce monitor semantics. CRM reports use the existing widget data runtime directly through `crm-reports.js` and the API summary endpoint. |
| Reports module | Seed grid-resident stat/chart/table widgets | The plan calls for reports as aggregates and builder widgets, so Phase 7 activates normal builder widgets keyed to CRM datasets instead of creating another overlay theater. |
| `/api/reports/summary` | Add server-side aggregate endpoint | Reports need shared team truth from Postgres/API. The endpoint groups tickets, deals, contacts, tasks, calendar items, history, and recent records without returning to MQTT or local-only state. |
| Card-system create action | Expose `create` / `openCreate` on each instance | Quick-add should reuse the same draft lifecycle, validation, flyout, and abandon behavior as the stack `+` buttons rather than adding a parallel modal flow. |
| Quick-add launcher | Global `+` delegates to active module factories | Creating a ticket, deal, or contact should work from any workspace while preserving module-specific card creation rules and backend writes. |
| API URL changes | Reconnect the generic store, clear caches, and refresh all entities | Switching team backends must not leave the old WebSocket or stale records alive. The Postgres/API store owns the shared truth, so changing its endpoint is a real backend switch. |
| Backend status UI | Add it to the existing account menu | Team polish needs a visible shared-backend status and API URL editor, but a new top-level chrome species is unnecessary. The account menu already owns user/team controls. |
| `CRM_VISION.md` continuation | Treat as Phase 8+ roadmap | The original `CRM_PLAN.md` ended at Phase 7; the vision document extends the same Postgres/API, card-system, and camera direction with Money, Today, automations, and deeper reporting. |
| Invoice entity | Add to the shared Postgres/API store | Invoices need shared cash state, due dates, history, versioning, and WebSocket broadcasts like every other CRM record; a renderer-only store would break team truth. |
| Money module | Instantiate through `card-system.js` / `card-detail.js` | Draft, Sent, Overdue, and Paid match the existing deck/bucket/resolved-pile choreography. Only small config hooks were added so bucket movement can update invoice `state` and Paid can bypass ticket-style stage completion. |
| Interaction entity | Add API-backed fan-out | Interactions are relationship events, not a standalone theater. Creating one on the server appends history and stamps `lastTouchAt` on related records so contact attention and reports use shared truth. |
| Invoice overdue nudge | Run on the API server | Overdue is derived from shared `dueDate`, so the minute sweep belongs near Postgres and the change stream, not in an Electron client that may be closed. |
